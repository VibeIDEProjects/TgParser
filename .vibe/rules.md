## Папка `.vibe/`

Базовые сценарии «что куда класть» (импорт правил из Cursor, цели, планы, workflows) IDE дополнительно подмешивает в системный контекст; здесь держите **только** специфику **этого** репозитория.

### Структура .vibe/

| Путь | Назначение |
|------|------------|
| `.vibe/rules/` | Правила для ИИ-агента (markdown-файлы, импортируются автоматически) |
| `.vibe/rules/dev-engine.mdc` | Правила разработки (создание моделей, миграции и т.п.) |
| `.vibe/rules/knowledge.mdc` | Правила ведения базы знаний |
| `.vibe/rules/release.mdc` | Правила релизов |
| `.vibe/rules/roadmap.mdc` | Правила ведения роадмапа |
| `.vibe/skills/` | Навыки (Agent Skills) — вызываются через `/skill:name` или авто-активируются по `description` |

## Папка `docs/`

Вся документация проекта хранится в `docs/`. Никакие `.md` файлы документации не создаются в корне проекта (кроме `README.md`).

| Файл | Назначение |
|------|------------|
| `docs/idea.md` | Идея проекта — сырая, неструктурированная |
| `docs/product.md` | Продуктовая спецификация — что делаем, сценарии, edge-cases |
| `docs/tech.md` | Техническая спецификация — архитектура, стек, модели |
| `docs/roadmap.md` | Пошаговый план разработки |
| `docs/knowledge.md` | База знаний проекта (инсайты, термины, договорённости) |

## Навыки (Agent Skills)

Навыки — это отдельные `.vibe/skills/<name>/SKILL.md` файлы с YAML-фронтматтером. Агент активирует навык когда задача совпадает с `description` или пользователь вызывает `/skill:<name>`.

### Сводка всех навыков

| Навык | Назначение | Когда применять |
|-------|-----------|-----------------|
| `spec-driven-implementation` | Дирижёр spec-first процесса: PRODUCT.md → TECH.md → реализация → верификация | Значительные фичи (1k+ LOC, кросс-стек, высокая неоднозначность) |
| `write-product-spec` | Создать `PRODUCT.md` — пользовательское поведение, инварианты, edge-cases, критерии успеха | Перед реализацией фичи, когда поведение неочевидно |
| `write-tech-spec` | Создать `TECH.md` — архитектура, границы модулей, трейдоффы, план тестирования | Когда реализация затрагивает несколько подсистем или есть риск регрессий |
| `implement-specs` | Реализация строго по утверждённым `PRODUCT.md` + `TECH.md`, спецификации и код в одном PR | После утверждения спек, шаг «строй» |
| `review-pr` | Структурированное ревью diff'а → `review.json` (не постит на GitHub) | Когда PR уже в локальных артефактах (`pr_diff.txt` + `pr_description.txt`) |
| `resolve-merge-conflicts` | Разрешение конфликтов через `extract_conflict_context.py` — без чтения целых файлов | При конфликтах после merge/rebase/cherry-pick/stash pop |
| `update-skill` | Создать или обновить `SKILL.md`: структура, фронтматтер, best-practices | Когда нужно добавить новый навык или поправить существующий |
| `example` | Демонстрационный навык для проверки `/skill:` | Тестирование механизма навыков |

### Типовой рабочий процесс (от идеи до ревью)

```
1. spec-driven-implementation  →  решает, нужны ли спеки
2. write-product-spec          →  PRODUCT.md (что делаем для пользователя)
3. write-tech-spec             →  TECH.md   (как делаем внутри)
4. implement-specs             →  код + тесты по спекам
5. review-pr                   →  review.json (проверка результата)
```

Навыки `resolve-merge-conflicts` и `update-skill` — вспомогательные, используются по мере необходимости.
