# Скиллы для вайбкодинга в большом проекте

Это набор из 7 универсальных скиллов для VibeIDE, собранный из открытой части кодовой базы Warp. Из репозитория убраны всё, что завязано на их конкретный стек (Rust, Warp UI, фреймворк интеграционных тестов и т.д.) и оставил то, что можно положить в любой проект — фронтенд, бэкенд, моно-репо, что угодно.

## Что внутри

```
skills/
├── spec-driven-implementation/   ← дирижёр процесса
├── write-product-spec/           ← PRODUCT.md: как должно работать
├── write-tech-spec/              ← TECH.md: как это реализовать
├── implement-specs/              ← как писать код по утверждённой спеке
├── review-pr/                    ← структурированное ревью в review.json
├── resolve-merge-conflicts/      ← разрешение конфликтов без чтения целых файлов
└── update-skill/                 ← мета-скилл: как писать новые скиллы
```

---

# Зачем это вообще нужно

В маленьком проекте можно «вайбкодить» — давать модели общие задачи и принимать что получилось. В большом так нельзя: код накапливается, паттерны размножаются, регрессии прячутся в углах. Через полгода у вас три способа делать одно и то же, тесты ничего не ловят, а каждая новая фича превращается в раскопки.

Скиллы — это способ зафиксировать процесс не в голове и не в Notion, а в самом репозитории, как файлы. Модель сама подтягивает нужный скилл в нужный момент по ключевым словам в запросе. По сути, вы превращаете «как мы тут работаем» в исполняемые инструкции для агента.

Дальше — как с этим всем жить.

---

# Жизненный цикл фичи

Семь скиллов выше — это не отдельные инструменты, а звенья одной цепочки. Логика такая:

```
идея → PRODUCT.md → TECH.md → код в том же PR → ревью → мердж
                                   ↑
                       (правим спеку по ходу дела)
```

## Шаг 0. Решить, нужна ли вообще спека

Главный принцип — **прагматизм**. Не каждое изменение требует документа.

Спека нужна, когда:
- задача большая, ориентировочно от тысячи строк кода
- затрагивает несколько модулей
- есть неоднозначность в продуктовом или техническом плане
- регрессия будет дорогой

Спека не нужна, когда:
- мелкий баг-фикс
- прямолинейный рефакторинг
- косметика в интерфейсе

Этим занимается скилл `spec-driven-implementation` — он работает дирижёром: смотрит на задачу и решает, какие из остальных скиллов вызывать.

## Шаг 1. PRODUCT.md — что должно работать

Скилл `write-product-spec` создаёт файл `specs/<номер-тикета>/PRODUCT.md`. Это не презентация и не PRD в духе «продуктовое видение». Это сухое перечисление поведения с точки зрения пользователя, **пронумерованными пунктами, каждый из которых можно проверить тестом**.

Пример (фрагмент):

> 1. Когда блок вывода содержит таблицу markdown, она рендерится как визуальная таблица, а не как сырой текст с pipe-символами.
> 2. Пустые ячейки рендерятся пустыми, той же высотой, что и соседние.
> 3. Если таблица шире блока, появляется горизонтальная прокрутка. Вертикальная прокрутка блока работает независимо.
> ...

Ключевая идея: пока такого списка нет, модель будет додумывать поведение в краевых случаях. После — она просто реализует пункты.

В скилле есть полный пример из 17 инвариантов про разные краевые случаи. Перечитайте его — это эталон того, как должна выглядеть секция Behavior.

## Шаг 2. TECH.md — как реализовать

`write-tech-spec` создаёт `specs/<номер-тикета>/TECH.md`. Перед написанием скилл явно требует **читать актуальный код**, а не сочинять архитектуру с нуля.

Структура жёсткая:
- **Context** — что меняем, как сейчас устроено, ссылки на конкретные файлы со строками: `src/workspace/mod.rs:42`
- **Proposed changes** — план реализации: какие модули меняются, какие новые типы появляются, как это вписывается в существующие паттерны
- **Testing and validation** — каждый пронумерованный пункт из PRODUCT.md ссылкой на конкретный тест

Тех-спека пишется не всегда. Если задача только про UI — может хватить и одной продуктовой. Если задача исследовательская — иногда проще сначала собрать прототип, а потом написать TECH.md «по итогам».

## Шаг 3. Код в том же PR

`implement-specs` — это правило: **спеки и код едут в одном пулл-реквесте**. Не «сначала смержим PRD, потом отдельно код». Один PR — одна фича, и спека там же.

Почему так:
- ревьюер видит, что реально отгрузили, а не что задумывали месяц назад
- если по ходу реализации поведение поменялось — правится PRODUCT.md прямо в этом же PR
- спеки не протухают, потому что они физически не могут разойтись с кодом

В скилле есть опциональные `PROJECT_LOG.md` и `DECISIONS.md` — для очень больших фич, чтобы не терять контекст между сессиями. По умолчанию они не нужны.

## Шаг 4. Ревью

`review-pr` — это формат ревью, заточенный под автоматизацию. Модель не пишет комментарии в GitHub напрямую, а формирует файл `review.json` со строгой структурой. Воркфлоу в CI потом раскидывает комментарии по файлам.

Каждый комментарий обязан начинаться с одной из меток:
- `🚨 [CRITICAL]` — баг, дыра в безопасности, потеря данных
- `⚠️ [IMPORTANT]` — логическая ошибка, упущенный краевой случай
- `💡 [SUGGESTION]` — стоящее улучшение
- `🧹 [NIT]` — мелочи, и **только если приложен блок-предложение** ```suggestion

Последнее правило — золотое. Оно отсекает 80% «ну тут можно было бы покрасивее» без конкретики.

## Шаг 5. Конфликты при мердже

`resolve-merge-conflicts` — самый необычный из семи. У него внутри лежит Python-скрипт, который **не открывает конфликтные файлы целиком**. Вместо этого он показывает только ours/base/theirs-куски и компактный diff между «нашими» и «чужими» правками.

Зачем: у модели ограниченный контекст. Если конфликт в файле на 5000 строк, читать весь файл — лишний расход. Скрипт выдаёт ровно то, что нужно для принятия решения.

# Как разработчики реально с этим работали

Несколько наблюдений по тому, как устроена их рабочая дисциплина:

**Спека хранится рядом с кодом, не в Confluence.** Папка `specs/<номер-тикета>/` лежит в самом репозитории и обновляется вместе с кодом. Это значит: история спеки — это история коммитов; «правда» о фиче — это то, что закоммичено сейчас, а не презентация трёхмесячной давности.

**Скиллы связаны между собой.** В конце почти каждого `SKILL.md` есть секция `## Related Skills` со ссылками на соседей. Когда модель доделывает PRODUCT.md, она автоматически узнаёт, что следующий шаг — `write-tech-spec`. Так получается граф, по которому агент движется сам, без вашей навигации.

**Описание скилла — это не «о чём оно», а триггер.** Поле `description` начинается с глагола в третьем лице и обязательно содержит фразу «Use when…» с ключевыми словами из реальных пользовательских запросов. Если этого нет — модель не найдёт скилл в нужный момент.

**Каждый скилл — короткий.** Большинство из них умещается в 80–150 строк. Если перевалило за 200 — отдельные куски выносятся в подпапку `references/`, и в основном файле остаются только ссылки. Так модель грузит в контекст ровно нужное, а не всё подряд.

**Длина регулируется по задаче.** В спеках жёстко прописано: тривиальное изменение — спеки вообще нет, маленькая фича — 30–60 строк, средняя — 80–150, большая — длиннее, но без воды. Главное: **раздувать только секцию Behavior**, всё остальное (Problem, Goals, Summary) держать тонким.

**«Pragmatic above all else».** Эта фраза стоит в нескольких скиллах как финальный пункт. Спеки нужны не для процесса ради процесса, а чтобы агент лучше делал свою работу. Если они мешают — их не пишут.

# Лайфхаки и неочевидное

**1. Структурированный вывод вместо свободного текста.**
`review-pr` пишет JSON со схемой, а не markdown. Это даёт возможность автоматически разбирать вывод и постить комментарии скриптом. Тот же подход работает для любого скилла, где результат должен куда-то уехать дальше: триаж тикетов, отчёт по CI, чек-листы.

**2. Скилл может приносить с собой утилиту.**
`resolve-merge-conflicts` тащит с собой Python-скрипт. Это важный паттерн: если для задачи нужен узкий инструмент, который экономит контекст — положите его прямо в папку скилла и ссылайтесь на него в инструкциях.

**3. Делать план, а не делать сразу.**
В скилле диагностики CI (которого здесь нет, но идея общая) прямо написано: «Никогда не правь код сразу. Сначала сформируй план, дай пользователю посмотреть.» Это правило стоит вынести в инструкции для любого скилла, где речь идёт о массовых правках в чужом коде.

**4. Не врать про спеку.**
Правило «спека описывает то, что реально отгрузили» — это, наверное, самое важное во всём процессе. Если по ходу реализации вы поменяли поведение, не приукрашайте спеку «как задумывалось» — правьте её под реальность. Иначе через год никто не будет ей доверять.

**5. Поле `description` — главное в SKILL.md.**
Модель не читает все SKILL.md подряд. Она читает только их `description` из frontmatter и решает по описанию, какой скилл подгрузить. Если описание расплывчатое — скилл не вызовется. Хорошее описание включает:
- глагол в начале («Resolves…», «Writes…», «Reviews…»)
- ключевые слова из реальных запросов («when the user asks for», «when X happens»)
- упоминание конкретных файлов/команд/концепций, по которым его можно опознать

**6. Specializes — расширение скиллов под конкретный репозиторий.**
В исходном репозитории были файлы вида `review-pr-local`, `triage-issue-local` с полем `specializes: review-pr` во frontmatter. Идея: общий скилл живёт глобально и описывает контракт (формат вывода, безопасность), а локальный — добавляет особенности конкретного проекта (свои метки, своя терминология, свои паттерны кода). Если будете масштабировать набор на много проектов — этот приём пригодится.

**7. Скиллы можно итерировать.**
В `update-skill/references/best-practices.md` есть раздел «Skill Iteration»: следите, где модель буксует на скилле, и точечно подкручивайте именно эти места. Не добавляйте предупреждающие правила «на всякий случай» — добавляйте только то, что в реальности выстрелило.

# Что брать первым делом

Если хочется проверить эту схему быстро, без полного развёртывания:

1. Скопируйте `update-skill/` — он покажет, как писать новые скиллы под ваш стек.
2. Скопируйте `resolve-merge-conflicts/` — он начнёт работать сразу, на любом git-репо.
3. Когда возьмётесь за первую крупную фичу — добавьте триаду `spec-driven-implementation` + `write-product-spec` + `write-tech-spec` + `implement-specs`.
4. `review-pr` подключайте, когда захотите автоматизировать ревью в CI.

И помните: скиллы — это про дисциплину, а не про магию. Они не сделают за вас архитектуру и не заменят ревью живым человеком. Они лишь гарантируют, что агент будет идти по одному и тому же маршруту, а не каждый раз заново изобретать процесс.
